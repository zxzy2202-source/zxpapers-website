# 网站整体检查报告（含后台）

检查时间：2026-06-29
项目：zxpapers-website
范围：前台页面、后台管理、API、鉴权、存储、图片、部署与构建

## 总体结论

网站当前代码可以通过类型检查和生产构建，前台页面与后台功能模块整体结构完整。最需要优先处理的是后台安全配置与部署数据层配置，而不是前台编译问题。

核心判断：

1. 构建通过：`tsc --noEmit` 和 `next build` 已通过。
2. 后台 API 普遍有鉴权保护。
3. 后台页面访问依赖 `middleware` 统一拦截，页面组件自身没有二次鉴权兜底。
4. 生产环境如果漏配后台密码或 session secret，会退回到硬编码默认值，属于高风险。
5. 图片、Hero、SEO、文章、询盘等后台数据依赖 KV/R2 与环境变量；如果线上 KV/R2 配置不完整，后台保存和图片显示会受影响。

## P0 高优先级问题

### 1. 后台存在默认密码与默认 session secret 回退

相关位置：

- `src/lib/auth.ts`
- `src/lib/authEdge.ts`

问题：

- `verifyPassword()` 在未配置 `ADMIN_PASSWORD` 时会使用默认密码 `zxpaper2026`。
- `getSecret()` 在未配置 `ADMIN_SESSION_SECRET` 时会回退到 `ADMIN_PASSWORD`，再回退到 `zxpaper-dev-secret`。
- 如果生产环境变量漏配，后台会变成已知默认密码和已知签名密钥。

风险：高。生产环境一旦漏配，后台存在被撞库或伪造 session 的风险。

建议修复：

- 生产环境中强制要求 `ADMIN_PASSWORD` 和 `ADMIN_SESSION_SECRET` 存在。
- 生产环境缺失时直接拒绝登录或抛出配置错误，不允许使用默认值。
- `ADMIN_SESSION_SECRET` 不应回退到 `ADMIN_PASSWORD`，两者应独立。

### 2. 后台页面鉴权只依赖 middleware，页面层无兜底

相关位置：

- `src/middleware.ts`
- `src/app/admin/layout.tsx`
- `src/app/admin/page.tsx`
- `src/app/admin/images/page.tsx`
- `src/app/admin/posts/page.tsx`
- `src/app/admin/inquiries/page.tsx`
- `src/app/admin/seo/page.tsx`

结论：

- 当前并不是完全裸奔，因为 `middleware.ts` 会拦截 `/admin` 路径。
- 但后台页面组件自身没有调用 `isAuthenticated()` 或 `redirect('/admin/login')`。
- 如果未来 middleware matcher 改坏、部署平台行为变化，页面层没有第二道保护。

建议修复：

- 在 `src/app/admin/layout.tsx` 增加服务端鉴权逻辑。
- 登录页单独排除。
- 保留 middleware，形成“双层保护”。

## P1 中高优先级问题

### 3. 后台数据强依赖 KV 配置，线上漏配会影响保存能力

相关位置：

- `src/lib/storage.ts`
- `src/lib/heroStore.ts`
- `src/lib/imageSlotStore.ts`
- `src/lib/postsStore.ts`
- `src/lib/inquiryStore.ts`
- `src/lib/seoStore.ts`

问题：

- `getStorage()` 只有同时存在 `KV_REST_API_URL` 和 `KV_REST_API_TOKEN` 时才走 Vercel KV。
- 否则退回本地 `data/*.json`。
- Vercel 等 Serverless 环境不适合依赖本地文件持久化。

影响范围：

- 首页 Hero 保存
- 图片管理保存
- SEO 设置保存
- 文章管理保存
- 询盘保存与状态更新
- 登录/询盘限流记录

建议检查线上环境：

- `KV_REST_API_URL`
- `KV_REST_API_TOKEN`
- KV 中是否存在 `zxp:hero`、`zxp:image-overrides`、`zxp:seo`、`zxp:posts`、`zxp:inquiries`

建议代码改进：

- 在生产环境缺少 KV 时显示明确错误，不要静默回退到本地文件。
- 后台首页增加“存储状态检查”。

### 4. R2 图片配置是图片显示和后台上传的关键依赖

相关位置：

- `src/lib/r2.ts`
- `src/app/api/admin/upload-r2/route.ts`
- `next.config.ts`

检查结果：

- 本地静态图如 `/images/logo-dark.png` 可正常访问。
- 多数前台图依赖 `/r2-assets/...` 或 CloudFront/R2。
- 之前已修复 3 个 OEM 页面空 fallback 问题。

仍需检查线上：

- `NEXT_PUBLIC_R2_URL`
- `R2_ACCOUNT_ID`
- `R2_ACCESS_KEY_ID`
- `R2_SECRET_ACCESS_KEY`
- `R2_BUCKET_NAME`
- R2 bucket 是否公开可读
- R2 自定义域或 `r2.dev` 公开 URL 是否仍可访问

建议改进：

- 后台图片管理页显示 R2 配置状态。
- 上传失败时提示具体缺哪个变量。
- 对坏链图片做前台兜底图展示。

## P2 中优先级问题

### 5. 文章 slug 编辑存在潜在重复风险

相关位置：

- `src/lib/postsStore.ts`

现状：

- 新建文章时会检查 slug 是否重复。
- 编辑文章时直接合并数据，没有检查新 slug 是否与其他文章重复。

风险：

- 管理员编辑文章 slug 时可能制造重复 URL。

建议修复：

- 编辑文章时，如果 slug 变化，应检查除自身外是否已有相同 slug。

### 6. 后台前端操作对 API 失败反馈不完整

相关位置：

- `src/app/admin/inquiries/InquiriesClient.tsx`
- `src/components/admin/PostEditor.tsx`
- `src/app/admin/images/ImageManagerClient.tsx`

现状：

- 图片管理错误提示相对完整。
- 询盘标记/删除没有检查接口返回状态，失败时 UI 可能仍更新。
- 文章删除没有检查接口返回状态。

建议修复：

- 所有后台写操作统一检查 `res.ok`。
- 失败时保留原 UI 状态并提示错误。

### 7. 构建有依赖警告但不影响当前上线

构建结论：

- `tsc --noEmit` 通过。
- `next build` 通过。
- Next.js 成功生成 118 个页面。

警告：

- `caniuse-lite` / Browserslist 数据较旧。
- `baseline-browser-mapping` 数据较旧。
- AWS SDK v3 提示未来版本会要求 Node >=22；当前 Node 20.12.0 仍满足项目 `package.json` 的 `20.x || 22.x || 24.x`。

建议：

- 短期不用阻塞上线。
- 下次维护时升级 Browserslist 数据。
- 中期建议部署环境升级到 Node 22。

## 已确认正常的部分

### 1. 构建与类型检查正常

- TypeScript 检查通过。
- Next.js 生产构建通过。
- 前台、后台、API 路由均能被构建系统识别。

### 2. 后台 API 普遍有鉴权

已看到以下 API 都调用了 `isAuthenticated()`：

- `/api/admin/hero`
- `/api/admin/images`
- `/api/admin/posts`
- `/api/admin/inquiries`
- `/api/admin/seo`
- `/api/admin/upload-r2`
- `/api/admin/indexnow`

### 3. 询盘接口有基础反垃圾能力

- `/api/inquiry` 有 IP 限流。
- 必填字段有基础校验。
- 邮箱格式有基础正则检查。
- 通知推送使用 `Promise.allSettled()`，单个渠道失败不会拖垮全部通知。

## 建议修复顺序

1. P0：移除生产默认密码与默认 secret 回退。
2. P0：给 `admin/layout` 增加页面级服务端鉴权兜底。
3. P1：生产环境缺 KV 时明确报错，不再静默退回本地文件。
4. P1：后台增加 KV/R2 配置状态检查卡片。
5. P1：检查并确认线上 Vercel 环境变量完整性。
6. P2：修复文章编辑 slug 重复问题。
7. P2：完善后台写操作失败提示。
8. P2：维护依赖数据与 Node 版本。

## 结论

网站整体可构建、可运行，前台结构完整，后台功能模块也较完整。当前最需要处理的是“生产后台安全”和“线上数据/图片存储配置可观测性”。建议先做 P0 两项安全修复，再做 KV/R2 状态检查，这样可以同时降低后台风险，并减少后续“图片不显示、后台保存不生效”的排查成本。
