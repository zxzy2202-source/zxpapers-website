# zxpapers 网站检查结论

## 检查范围
- 技术栈与结构：`package.json`、`src/app`、`src/components`
- 核心页面：首页、全局布局、Header、Footer、PageHero
- 检查方式：当前工作区源码核对，不依赖线上截图

## 总体判断
这个站的工程底子是好的：Next.js 15 + App Router 结构清晰，首页模块化程度高，`metadata`、`Organization/WebSite` 结构化数据、`PageHero` 组件、`Skip to main content`、`prefers-reduced-motion` 等都做得比较完整。

当前主要问题不在“有没有内容”，而在：
1. 全局布局有几个硬编码耦合点；
2. 桌面/平板导航体验还有明显风险；
3. 首页信息密度偏高、重复偏多，影响转化节奏；
4. 有少量代码卫生与语义细节待清理。

## 当前问题清单

### P1｜Header 高度与页面顶部留白强耦合
**现象**
- Header 是固定定位：`src/components/layout/Header.tsx:152`
- 主导航固定高度 `h-[68px]`：`src/components/layout/Header.tsx:177`
- `main` 通过硬编码 `pt-[98px] sm:pt-[92px]` 让内容避开 Header：`src/components/layout/Layout.tsx:20`

**风险**
- 只要顶部条高度、字体、换行或导航高度有一点变化，页面首屏就可能出现“内容被盖住”或“顶部多空白”。
- 这是全站性风险，不是单页问题。

**建议**
- 优先用单一 CSS 变量管理 Header 高度，Header 和 `main` 共同引用。
- 或把 `fixed` 改为更适合当前结构的 `sticky`，减少补偿魔数。

### P1｜Products 下拉菜单存在右侧溢出风险
**现象**
- Products / 普通 mega menu 的最小宽度最大到 `920px`：`src/components/layout/Header.tsx:245`
- 只有 Markets 菜单做了居中修正：`src/components/layout/Header.tsx:247`
- Products 默认仍是 `left: 0` 展开：`src/components/layout/Header.tsx:243`

**风险**
- 当触发项靠近导航右侧时，下拉面板会超出视口，影响桌面端导航可用性。

**建议**
- 给 Products 菜单也加居中/右对齐保护。
- 同时加 `max-width: calc(100vw - 2rem)` 一类的视口约束。

### P1｜平板端被强制降级为汉堡菜单
**现象**
- 桌面导航只在 `lg` 以上显示：`src/components/layout/Header.tsx:196`

**风险**
- 768–1023px 的平板设备看不到水平导航，采购型用户需要多一步点击才能进入 Products/Markets，信息发现效率下降。

**建议**
- 至少评估把断点从 `lg` 下探到 `md`。
- 如果担心空间不够，可做一个平板简化版导航，而不是完全汉堡化。

### P2｜首页纵向偏长，节奏仍然偏重
**现象**
- 首页存在大量 `py-20` 段落：`src/app/page.tsx:381`、`src/app/page.tsx:452`、`src/app/page.tsx:505`、`src/app/page.tsx:605`、`src/app/page.tsx:872`
- 深色段也较多：`src/app/page.tsx:349`、`src/app/page.tsx:561`、`src/app/page.tsx:605`、`src/app/page.tsx:731`

**风险**
- 页面滚动成本高，视觉重音过多，核心 CTA 容易被稀释。
- B2B 首页更应该尽快把用户推向“产品分类 / 定制能力 / 询盘入口”。

**建议**
- 给 section 建立 `py-12 / py-16 / py-20` 分层，而不是多个模块统一 `py-20`。
- 深色区块控制在 2 个重音位更合适。

### P2｜工厂信任信息重复较多
**现象**
- 首页 trust strip 一次：`src/app/page.tsx:349`
- 首页工厂数据四宫格又一次：`src/app/page.tsx:564`
- Jumbo/物流文案又继续复用国家数等数据：`src/app/page.tsx:745`
- Key Facts 区再次输出：`src/app/page.tsx:835`
- Footer 文案也重复工厂资质/年限：`src/components/layout/Footer.tsx:59`

**风险**
- 用户会更快“看腻”，但对产品差异、能力边界、MOQ、交付条件的理解并不会同步增强。

**建议**
- 保留 2 处最强信任表达：首屏简版 + 页面中后段详版。
- 其他位置改成“交付能力 / 案例 / SKU / 采购场景”类信息。

### P2｜首页部分 section 缺语义锚点
**现象**
- 有些 section 带 `aria-labelledby`，如：`src/app/page.tsx:381`、`src/app/page.tsx:436`
- 也有一些 section 没带，例如：`src/app/page.tsx:505`、`src/app/page.tsx:561`、`src/app/page.tsx:731`、`src/app/page.tsx:872`

**风险**
- 对屏幕阅读器、文档结构清晰度和长期语义维护都不够统一。

**建议**
- 给所有核心 section 补齐 heading id + `aria-labelledby`。

### P3｜Footer 存在代码卫生问题
**现象**
- 注释中有乱码：`src/components/layout/Footer.tsx:15`、`src/components/layout/Footer.tsx:39`、`src/components/layout/Footer.tsx:279`
- Footer 主体在 `md` 断点是 `2` 列，在 `lg` 才到 `6` 列：`src/components/layout/Footer.tsx:41`

**风险**
- 乱码虽然不影响页面渲染，但说明文件编码/拷贝历史不干净。
- 中屏宽度下链接墙会偏长，阅读节奏一般。

**建议**
- 先清理乱码注释。
- 再评估 `md:grid-cols-3` 是否比当前 `md:grid-cols-2` 更平衡。

## 优先级建议

### 先处理（高收益、低改动）
1. Header 高度解耦
2. Products mega menu 溢出保护
3. 平板导航断点优化
4. Footer 乱码清理

### 第二批（提升转化与信息效率）
5. 首页 spacing 分层
6. 精简重复 trust 信息
7. 减少深色区块数量
8. 补齐缺失的 section 语义关联

---

## 简短结论
这个网站不是“做得差”，而是已经到了该做第二阶段优化的时候：基础工程、SEO 与模板化都打好了，但首页节奏、导航体验和全局布局耦合点开始成为下一步增长瓶颈。

如果目标是**提高询盘转化与可维护性**，我建议先修导航与布局耦合，再压缩首页信息冗余。
