旧总览仍写“145 份正式文档”，不能继续作为实时规模依据。新库应由自动盘点或索引清单生成数量，不手工维护总数。

## 2. 旧结构的主要问题

1. 产品线成熟度不一致，却套用相似目录名称，导致目录看似统一、实际可用性不同。
2. 汇总报告、总索引、检查报告和正式正文多处并列，日常入口不唯一。
3. 豪正分支体量占整库约 60%，直接照搬会让其他产品线失去可读性。
4. 热敏标签和 NCR 的产品线根入口正文很弱，用户必须依赖目录猜测阅读路径。
5. 研究、销售工具、网站运营和证据治理混在产品目录内，跨品类复用成本高。
6. 手工统计已失真，缺少基于真实节点和状态的持续盘点。

## 3. 分解规则

旧文档不按原目录整体搬迁，而按“主用途”唯一归属：

| 旧内容模式 | 新归属 | 处理 |
|---|---|---|
| 产品知识、规格、选型、材料、工艺 | `10-产品知识包` | 提炼为稳定事实和矩阵 |
| A/B/C、市场、行业、区域、画像、JTBD、ICP | `20-市场与客户` | 去重后保留唯一正文 |
| D/E、话术、资格、报价、打样、试机 | `30-销售与交付` | 改成任务型工具 |
| 页面、URL、Brief、SEO、GEO、发布复盘 | `40-网站与增长` | 与当前网站路由绑定 |
| 声明、法规、证书、测试、设备、案例 | `50-证据与合规` | 统一 Claim ID 和状态 |
| F、治理、同步、检查模板 | `90-治理与模板` | 保留规则，移除重复业务结论 |
| 原始报告、旧稿、未采用研究 | `99-旧库分解` 或外部 Archive | 保留溯源，不进入日常入口 |

## 4. 迁移批次

### P0：先支撑询盘与现有页面

- 四条产品线索引。
- 规格/适配/报价字段。
- ICP、线索资格、报价前问询、打样/试机确认。
- 当前网站 URL Owner、页面 Brief、关键 FAQ。
- 对外声明与证据登记入口。

### P1：补齐成交和增长闭环

- 行业与买家角色。
- 痛点到方案、异议处理、交付 SOP。
- SEO/GEO 台账、内链、发布与复盘。
- 证书、测试和供应商文件索引。

### P2：吸收长期研究资产

- 国家/区域研究。
- 原子画像和长尾应用。
- 历史报告、竞品和原始底稿。

## 5. 不迁移或只保留链接

- 只有标题、没有稳定使用任务的空入口。
- 与正式正文重复的检查报告摘要。
- 手工维护的总文档数量。
- 未说明来源、Owner、状态和回写位置的泛分析。
- 仅替换行业名、国家名或产品名的模板化重复内容。

## 6. 飞书执行边界

本文件只是迁移设计，不代表已在飞书执行。远端阶段应按以下顺序单独确认：

1. 创建目标一级节点和产品知识包入口。
2. 先迁移 P0 的小批量文档并验证链接、正文和节点路径。
3. 建立旧节点到新节点的映射表。
4. 完成整批验证后，再决定旧节点保留为只读、归档或继续双轨运行。
5. 不删除旧节点，不直接重命名旧节点，不在未确认范围内批量移动。
